# Отчет о реализации

## 1. Цель проекта

Реализован MVP Telegram-бота на n8n для подготовки LinkedIn-контента под B2B-аудиторию в сфере game development, outsourcing и co-development.

Бот ведет пользователя от запроса идеи до утвержденного поста:

1. генерирует 12 тем;
2. дает выбрать тему;
3. спрашивает язык поста;
4. генерирует текст LinkedIn-поста;
5. генерирует 3 варианта изображения;
6. позволяет выбрать картинку или сгенерировать новые варианты;
7. принимает правки текста и картинки;
8. показывает финальный предпросмотр;
9. сохраняет результат как approved post.

В текущем MVP фактическая публикация в LinkedIn не встроена. Кнопка **Публиковать** сохраняет результат как утвержденный и готовый к публикации. Архитектура позволяет добавить публикационный слой отдельно.

## 2. Архитектура

```text
Telegram Bot
→ n8n workflow
→ PostgreSQL
→ LLM API
→ Image Generation API
→ Server file storage
→ RSS / trusted sources / LinkedIn-related source corpus
```

Сценарий сделан stateful. Это значит, что бот не просто вызывает LLM один раз, а хранит состояние пользователя, выбранную тему, язык, текущий текст, картинки, правки и финальный результат.

## 3. Пользовательский сценарий

```text
/start
→ Create post
→ Generate 12 topics
→ Select topic
→ Select language
→ Generate post
→ Generate 3 images
→ Select image
→ Final preview
→ Revise text / revise image / approve
```

Поддержаны повторные итерации:

- **Еще темы** — новая партия из 12 тем с учетом уже показанных тем.
- **Еще картинки** — новый batch из 3 изображений для текущего поста.
- **Исправить текст** — новая версия текста с учетом комментария пользователя.
- **Исправить картинку** — новые визуальные направления с учетом комментария пользователя.

## 4. Источники тем

### Интернет-источники

Для интернет-тем используется подход RSS / trusted sources. Материалы фильтруются по ключевым словам, связанным с game development и B2B-услугами:

- production
- pipeline
- co-development
- outsourcing
- live ops
- QA / testing
- art production
- Unity / Unreal
- cost / risk / quality / timelines

LLM не пересказывает новости, а превращает исходные сигналы в практические B2B-темы для LinkedIn-поста.

### LinkedIn-релевантные сигналы

Прямой масштабный сбор LinkedIn-контента чувствителен к правилам платформы. Поэтому выбран более безопасный подход:

- заранее заданный curated-пул публичных LinkedIn-страниц компаний;
- отдельный refresh-процесс;
- очистка текста;
- разбиение текста на chunks;
- сохранение chunks в PostgreSQL;
- ротация источников и балансировка по компаниям / категориям.

Основной Telegram-сценарий не парсит LinkedIn во время пользовательского запроса. Он берет уже подготовленный корпус из базы. Это ускоряет работу и повышает стабильность.

Для больших объемов источников архитектуру можно расширить через embeddings и RAG.

## 5. Управление состоянием

Карта состояний:

```text
idle
→ generating_topics
→ choosing_topic
→ choosing_language
→ generating_post
→ choosing_image
→ preview
→ revising_text
→ revising_image
→ approved
```

PostgreSQL хранит:

- текущий state пользователя;
- выбранную тему;
- выбранный язык;
- уже показанные темы;
- актуальный текст поста;
- версии текста;
- показанные изображения;
- выбранную картинку;
- комментарии к правкам;
- финальный approved post.

## 6. Антидубли и устойчивость

Антидубли реализованы через хранение уже показанных тем и передачу этого списка в prompt при следующей генерации.

Также предусмотрены fallback-подходы:

- если LinkedIn-источники недоступны — можно использовать последний сохраненный корпус;
- если RSS-источник не дал релевантных материалов — можно использовать fallback на broader internet signals;
- если пользователь пишет не по сценарию — бот ориентируется на текущий state и предлагает понятное действие.

Для production-версии стоит добавить semantic duplicate check через embeddings.

## 7. Логирование

В MVP основная история сохраняется в бизнес-таблицах: темы, посты, изображения, approved posts.

Для production-версии предусмотрен отдельный слой логирования:

- шаг workflow;
- входные данные;
- ответ внешнего сервиса;
- текст ошибки;
- session_id;
- timestamp.

## 8. Масштабирование

Архитектура позволяет добавить:

- очередь публикаций;
- календарь публикаций;
- мультиаккаунтность;
- несколько брендов / страниц;
- tone-of-voice профили;
- RAG для больших корпусов источников;
- S3 / Cloudflare R2 / CDN для изображений;
- админ-панель для утвержденных постов.

## 9. Итог

Реализован рабочий MVP n8n workflow, который можно проверить через Telegram-бота. Он закрывает ключевые требования: stateful workflow, повторные итерации, генерация тем/постов/изображений, правки, хранение контекста и подготовка архитектуры к будущей публикации в LinkedIn.
